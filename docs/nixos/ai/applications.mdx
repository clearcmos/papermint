## smart-commit

### Overview
AI-powered Git commit message generator that analyzes code changes and creates conventional commit messages.

### Configuration
- **Location**: `/home/nicholas/git/mine/smart-commit`
- **Config**: `/home/nicholas/.config/smart-commit/config.json`
- **Model**: qwen2.5-coder:7b-instruct
- **Endpoint**: `http://localhost:11434/api/generate`

### Current Configuration
```json
{
  "ai": {
    "api_url": "http://localhost:11434",
    "model": "qwen2.5-coder:7b-instruct",
    "backend_type": "ollama",
    "timeout": 120,
    "max_retries": 3
  }
}
```

### Usage Examples
```bash
# Basic usage
smart-commit

# With custom message
smart-commit -m "Custom prefix"

# Atomic commits (one per file)
smart-commit --atomic

# Test configuration
smart-commit test --all
```

### API Integration
```python
# smart-commit uses Ollama API
response = requests.post(
    "http://localhost:11434/api/generate",
    json={
        "model": "qwen2.5-coder:7b-instruct",
        "prompt": commit_prompt,
        "stream": False,
        "options": {
            "temperature": 0.3,
            "num_predict": 50
        }
    }
)
```

### Performance Characteristics
- **Response time**: 2-5 seconds for typical diffs
- **Model loading**: ~2 seconds if not already loaded
- **Accuracy**: High-quality commit messages for code changes
- **Context handling**: Excellent for large diffs (up to 8K tokens)

### Troubleshooting smart-commit

## keeper

### Overview
Google Keep API library with AI-powered note categorization and title generation.

### Configuration
- **Location**: `/home/nicholas/git/mine/keeper`
- **Models**: qwen2.5:7b-instruct (categorization), bge-m3 (embeddings)
- **Endpoints**: `http://localhost:11434/api/generate`, `/api/embeddings`

### API Integration
```python
# keeper AI service configuration
class KeeperAI:
    def __init__(self, 
                 categorization_url="http://localhost:11434",
                 embeddings_url="http://localhost:11434"):
        self.categorization_url = categorization_url
        self.embeddings_url = embeddings_url
```

### Core Features

**Note Categorization**:
```python
# Generate category suggestions
response = requests.post(
    f"{self.categorization_url}/api/generate",
    json={
        "model": "qwen2.5:7b-instruct",
        "prompt": f"Categorize this note: {content}",
        "stream": False,
        "options": {
            "num_predict": 100,
            "temperature": 0.2
        }
    }
)
```

**Title Generation**:
```python
# Generate smart titles for untitled notes
response = requests.post(
    f"{self.categorization_url}/api/generate",
    json={
        "model": "qwen2.5:7b-instruct", 
        "prompt": f"Generate title for: {content}",
        "stream": False,
        "options": {
            "num_predict": 50,
            "temperature": 0.3
        }
    }
)
```

**Semantic Embeddings**:
```python
# Generate embeddings for semantic search
response = requests.post(
    f"{self.embeddings_url}/api/embeddings",
    json={
        "model": "bge-m3",
        "prompt": content
    }
)
```

### Usage Examples

### Performance Characteristics
- **Categorization**: 1-3 seconds per note
- **Title generation**: 1-2 seconds per note
- **Embedding generation**: 0.5-1 second per note
- **Batch processing**: Efficient for multiple notes
- **Model switching**: Automatic between categorization and embeddings

## Audio Separator (UVR5)

### Overview
GPU-accelerated audio source separation using Ultimate Vocal Remover (UVR5) models with ROCm support. Separates vocals, instrumentals, drums, bass, and other stems from audio files.

### Configuration
- **Location**: `/var/lib/audio-separator` (venv and models)
- **Technology**: PyTorch ROCm + MDX-Net/Demucs models
- **GPU**: AMD RX 6800 XT via ROCm/HIP
- **Module**: `/etc/nixos/modules/desktop/ai/audio-separator.nix`

### Usage Examples
```bash
# Basic vocal removal (outputs vocals + instrumental)
uvr input.wav

# Use specific model for karaoke
uvr input.wav --model_name UVR-MDXNET_KARA_2

# Output only instrumental track
uvr input.wav --single_stem instrumental

# Output only vocals
uvr input.wav --single_stem vocals

# List available models
uvr-models

# Show all options
uvr-help
```

### Available Models

**MDX-Net Models** (fast, high quality):
| Model | Best For |
|-------|----------|
| `UVR-MDX-NET-Inst_HQ_3` | Instrumental extraction (default) |
| `UVR-MDXNET_KARA_2` | Karaoke creation |
| `UVR-MDX-NET-Voc_FT` | Vocal isolation |

**Demucs Models** (slower, multi-stem):
| Model | Stems |
|-------|-------|
| `htdemucs` | Vocals, drums, bass, other |
| `htdemucs_ft` | Fine-tuned version |

### Output Files
```bash
# Default output location: same directory as input
input.wav → input_(Vocals)_model.wav
          → input_(Instrumental)_model.wav

# Custom output directory
uvr input.wav --output_dir /path/to/output
```

### Performance Characteristics
- **Processing time**: ~10-30 seconds per minute of audio (GPU)
- **VRAM usage**: 2-6GB depending on model
- **Quality**: Professional-grade separation
- **Formats**: WAV, MP3, FLAC, OGG (via ffmpeg)

### ROCm Integration
The module uses pip-installed PyTorch from AMD's ROCm wheel index for reliable GPU acceleration:
- HSA_OVERRIDE_GFX_VERSION set for RX 6800 XT compatibility
- HIP_VISIBLE_DEVICES configured in amd-gpu.nix
- Automatic venv setup on first run

### Troubleshooting
```bash
# Check GPU is detected
python3 -c "import torch; print(torch.cuda.is_available())"

# Check ROCm
rocm-smi

# Reinstall if issues
sudo rm -rf /var/lib/audio-separator/venv
uvr --help  # Triggers reinstall

# Check available models
uvr-models
```

## ollama-websearch

### Overview
Python library and CLI for web-search-augmented Ollama queries. Combines SearXNG metasearch with Ollama LLM to provide grounded, citation-backed responses. Designed for programmatic use in automated workflows.

### Configuration
- **Module**: `/etc/nixos/packages/ollama-websearch.nix`
- **Python module**: `/etc/ollama-websearch/ollama_websearch.py`
- **Backend**: SearXNG (localhost:8280) + Ollama (localhost:11434)
- **Default Model**: qwen2.5:7b-instruct

### CLI Usage
```bash
# Basic usage
ollama-websearch "What is NixOS flakes?"

# With environment variables
OLLAMA_MODEL=qwen2.5:7b-instruct MAX_RESULTS=5 ollama-websearch "query"

# Available environment variables:
# OLLAMA_MODEL   - Model to use (default: qwen2.5:7b-instruct)
# SEARXNG_URL    - SearXNG endpoint (default: http://localhost:8280)
# OLLAMA_URL     - Ollama endpoint (default: http://localhost:11434)
# MAX_RESULTS    - Number of search results (default: 5)
```

### Python API
```python
# Load the module
exec(open('/etc/ollama-websearch/ollama_websearch.py').read())

# Main function: search + LLM response
answer = search_and_ask("What is NixOS?")
print(answer)

# Streaming version (real-time output)
for chunk in search_and_ask_stream("What is NixOS?"):
    print(chunk, end="", flush=True)

# Use components separately
results = web_search("NixOS flakes", max_results=5)
# Returns list of SearchResult(title, url, content, engine)

answer = ask_with_context(
    question="Explain flakes",
    context=results,  # or a string
    model="qwen2.5:7b-instruct"
)

# Direct Ollama call (no web search)
response = ask_ollama("Hello, world!", model="qwen2.5:7b-instruct")
```

### Available Functions

| Function | Purpose | Returns |
|----------|---------|---------|
| `web_search(query, max_results=5)` | Search via SearXNG | `list[SearchResult]` |
| `ask_ollama(prompt, model=...)` | Direct LLM query | `str` |
| `ask_with_context(question, context)` | LLM with provided context | `str` |
| `search_and_ask(query)` | Combined search + LLM | `str` |
| `search_and_ask_stream(query)` | Streaming version | `Generator[str]` |
| `format_results(results)` | Format SearchResults as string | `str` |

### SearchResult Dataclass
```python
@dataclass
class SearchResult:
    title: str      # Page title
    url: str        # Source URL
    content: str    # Snippet/description
    engine: str     # Search engine that returned it
```

### Performance Characteristics
- **Time to first token**: ~1.5 seconds
- **Total response time**: 6-9 seconds
- **Tokens/sec**: ~52 tok/s (native Ollama speed)
- **Search latency**: &lt;500ms (SearXNG)

### Integration Examples

**Automated Script**:
```python
#!/usr/bin/env python3
exec(open('/etc/ollama-websearch/ollama_websearch.py').read())

def research_topic(topic):
    """Research a topic and save results."""
    answer = search_and_ask(f"Summarize recent developments in {topic}")
    with open(f"{topic.replace(' ', '_')}.md", "w") as f:
        f.write(f"# {topic}\n\n{answer}\n")
    return answer

research_topic("NixOS flakes")
```

**With Custom System Prompt**:
```python

answer = search_and_ask(
    "What are the security best practices for NixOS?",
    system_prompt="You are a security expert. Provide detailed, actionable advice.",
    max_results=8,
    temperature=0.2
)
```

## Development and Customization

### Adding New Applications

**Integration Steps**:
1. **Choose appropriate model** based on task requirements
2. **Implement Ollama API calls** using standard HTTP requests
3. **Configure model-specific parameters** for optimal performance
4. **Test with existing infrastructure** to ensure compatibility

**API Standards**:
```bash
# Standard Ollama API format
curl -X POST http://localhost:11434/api/generate \
  -H "Content-Type: application/json" \
  -d '{
    "model": "model-name",
    "prompt": "your prompt",
    "stream": false,
    "options": {
      "temperature": 0.3,
      "num_predict": 100
    }
  }'
```

### Custom Application Development

**Best Practices**:
1. **Use streaming** for real-time feedback
2. **Implement timeouts** (30-120 seconds)
3. **Handle model loading delays** gracefully
4. **Cache responses** when appropriate
5. **Implement retry logic** for network issues

**Error Handling**:
```python
# Robust error handling example
try:
    response = requests.post(api_url, json=request_data, timeout=120)
    response.raise_for_status()
    return response.json()
except requests.exceptions.Timeout:
    # Handle timeout gracefully
    return {"error": "Request timed out"}
except requests.exceptions.ConnectionError:
    # Handle connection issues
    return {"error": "Cannot connect to AI service"}
```
