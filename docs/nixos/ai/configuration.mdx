## Service Configuration

### Location
- **Main config**: `/etc/nixos/modules/desktop/ollama.nix`
- **Host config**: `/etc/nixos/hosts/cmos/ai-config.nix`
- **Core integration**: `/etc/nixos/modules/desktop/ai-core.nix`

### Basic Service Setup

```nix
# /etc/nixos/modules/desktop/ollama.nix
services.ollama = {
  enable = true;
  
  # GPU acceleration
  acceleration = "rocm";
  package = pkgs.ollama-rocm;
  rocmOverrideGfx = "10.3.0";  # gfx1030 for RX 6800 XT
  
  # Network configuration
  host = "127.0.0.1";
  port = 11434;
  openFirewall = true;
};
```

### OLLAMA_KEEP_ALIVE Configuration

Controls how long models stay loaded in VRAM after the last request. This significantly impacts response latency.

**Valid Values:**

| Value | Behavior | Use Case |
|-------|----------|----------|
| `"-1"` | **Never unload** - Keep model in VRAM permanently | Instant responses, like llama.cpp behavior |
| `"0"` | **Immediate unload** - Remove from VRAM right after response | Minimal VRAM usage, max warmup delay |
| `"5m"` | **Default** - Unload after 5 minutes of inactivity | Balanced approach |
| `"30m"` | Keep loaded for 30 minutes | Good for active work sessions |
| `"24h"` | Keep loaded all day | Near-instant responses during workday |
| `"1m"` | Aggressive unloading after 1 minute | Gaming-optimized, frees VRAM quickly |

**Format Options:**
- Duration strings: `"10m"`, `"24h"`, `"90s"`
- Numeric seconds: `300` (equals 5 minutes)
- Negative for permanent: `"-1"` or `"-1m"`

**Performance Trade-offs:**

```nix
# Instant responses (like llama.cpp) - uses most VRAM
OLLAMA_KEEP_ALIVE = "-1";

# Balanced - good for active use
OLLAMA_KEEP_ALIVE = "30m";

# Gaming-optimized - frees VRAM quickly
OLLAMA_KEEP_ALIVE = "1m";
```

**Current Configuration:** The system uses `"1m"` for gaming compatibility, freeing VRAM quickly for KDE/games. This causes ~2-5 second warmup when re-loading models from disk.

**Note:** Individual API requests can override this with the `keep_alive` parameter.

### Multi-Model Configuration

When using multiple models, `OLLAMA_MAX_LOADED_MODELS` and `OLLAMA_KEEP_ALIVE` interact critically:

**Scenario 1: Single Model with Instant Response**
```nix
OLLAMA_MAX_LOADED_MODELS = "1";      # Only 1 model in VRAM
OLLAMA_KEEP_ALIVE = "-1";            # Never unload
```
- **Behavior**: First model loaded stays in VRAM indefinitely
- **Switching models**: Unloads current model, loads new one (warmup delay)
- **Best for**: Single-model workflows, minimal VRAM usage

**Scenario 2: Two Models with Instant Switching**
```nix
OLLAMA_MAX_LOADED_MODELS = "2";      # 2 models in VRAM
OLLAMA_KEEP_ALIVE = "-1";            # Never unload
```
- **Behavior**: Up to 2 models stay loaded, instant switching between them
- **Switching to 3rd model**: LRU (least recently used) gets unloaded
- **VRAM usage**: ~10-14GB (two 7B models)
- **Best for**: Workflows using 2 main models (e.g., coder + general)

**Scenario 3: Multiple Models with Timed Unload**
```nix
OLLAMA_MAX_LOADED_MODELS = "2";      # 2 models in VRAM
OLLAMA_KEEP_ALIVE = "30m";           # Unload after 30 min
```
- **Behavior**: Up to 2 models loaded, auto-cleanup after inactivity
- **Switching between 3+ models**: Recent models stay loaded, old ones unload
- **VRAM usage**: Scales down during idle periods
- **Best for**: Workflows using 3+ models throughout the day

**VRAM Sizing Guide (RX 6800 XT - 16GB Total):**

| Models | 7B VRAM Each | Total Used | KDE Reserve | Config |
|--------|--------------|------------|-------------|--------|
| 1 model | ~6GB | 6GB | 10GB free | `MAX_LOADED_MODELS = "1"` |
| 2 models | ~6GB | 12GB | 4GB free | `MAX_LOADED_MODELS = "2"` (recommended) |
| 3 models | ~6GB | 18GB | ❌ Exceeds VRAM | Not safe with 16GB |

**Recommended Configurations:**

```nix
# Gaming priority - minimize VRAM, accept warmup
OLLAMA_MAX_LOADED_MODELS = "1";
OLLAMA_KEEP_ALIVE = "30m";

# Productivity priority - instant switching between 2 main models
OLLAMA_MAX_LOADED_MODELS = "2";
OLLAMA_KEEP_ALIVE = "-1";

# Balanced - 2 models with auto-cleanup
OLLAMA_MAX_LOADED_MODELS = "2";
OLLAMA_KEEP_ALIVE = "1h";
```

## Model Management

### Automatic Model Download

```nix
# Pre-download service configuration
systemd.services.ollama-model-preload = {
  description = "Pre-download Ollama models if they don't exist";
  after = [ "network-online.target" "ollama.service" ];
  wantedBy = [ "multi-user.target" ];
  
  serviceConfig = {
    Type = "oneshot";
    RemainAfterExit = true;
    Environment = [
      "HOME=/var/lib/ollama"
      "OLLAMA_HOST=http://127.0.0.1:11434"
    ];
  };
};
```

### Required Models

The system automatically downloads these models:

1. **qwen2.5-coder:7b-instruct**
   - Purpose: Coding assistance (smart-commit)
   - Size: ~4.7GB
   - Quantization: Q4_K_M

2. **qwen2.5:7b-instruct**
   - Purpose: General knowledge tasks (keeper, sage)
   - Size: ~4.7GB
   - Quantization: Q4_K_M

3. **bge-m3**
   - Purpose: Text embeddings (keeper)
   - Size: ~1.1GB
   - Type: Embedding model


## Customization Options

### Performance Tuning

To modify performance settings, edit the `environmentVariables` section:

```nix
# For maximum performance (uses more VRAM)
OLLAMA_MAX_LOADED_MODELS = "3";      # Allow all models
OLLAMA_NUM_PARALLEL = "8";           # More parallel requests

# For maximum gaming compatibility (uses less VRAM)
OLLAMA_MAX_LOADED_MODELS = "1";      # Only one model at a time
OLLAMA_KEEP_ALIVE = "1m0s";          # Faster unloading
```

### Model Selection

To add or change models, modify the preload service:

```nix
# In ollama-model-preload service
MODELS=(
  "qwen2.5-coder:7b-instruct"
  "qwen2.5:7b-instruct"
  "bge-m3"
  # Add new models here
  "new-model:version"
)
```

## Troubleshooting Configuration

### Common Issues

1. **Service won't start**: Check ROCm installation and GPU access
2. **Models not downloading**: Verify network connectivity and disk space
3. **Poor performance**: Review GPU memory usage and thread allocation
4. **API not responding**: Check firewall rules and service binding

### Debug Mode

Enable debug logging:

```nix
# Add to environmentVariables
OLLAMA_DEBUG = "1";           # Enable debug logging
```

View debug output:

```bash
journalctl -fu ollama.service | grep -i debug
```
