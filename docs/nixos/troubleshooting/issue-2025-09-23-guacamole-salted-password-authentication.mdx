## Issue Summary

Apache Guacamole authentication was failing when using salted passwords in the PostgreSQL database. Unsalted passwords worked correctly, but any attempt to use the proper salted password hashing resulted in "Invalid login" errors.

## Root Cause Analysis

**Primary Issue**: PostgreSQL string escaping was modifying the password during hash computation in the NixOS module's systemd service script.

**Technical Details**:
1. **String Escaping Problem**: Using regular string literals in PostgreSQL caused escaping of special characters
2. **Algorithm Verification**: Apache Guacamole uses `password + UPPERCASE_HEX(salt)` for SHA-256 input
3. **Bash Heredoc Variable Substitution**: Required proper handling of variable expansion combined with PostgreSQL E string literals

## Technical Investigation

### Authentication Mechanism Verified

### Algorithm Research
Web research confirmed Apache Guacamole's exact password hashing specification:
- Salt concatenation order: `password + uppercase_hex_salt`
- Hashing process: `SHA-256(UTF-8_bytes(concatenated_string))`
- Salt encoding: Base16 (hex) with uppercase characters

### Hash Computation Testing
Step-by-step verification revealed discrepancies between expected and stored hashes, leading to discovery of the PostgreSQL string escaping issue.

## Solution Implemented

### Working Algorithm Implementation
```sql
-- Correct Guacamole password hashing in PostgreSQL
WITH salt_data AS (
  SELECT '\xFE24ADC5E11E2B25288D1704ABE67A79E342ECC26064CE69C5B3177795A82264'::bytea as salt_bytes
)
INSERT INTO guacamole_user (entity_id, password_hash, password_salt, password_date)
SELECT 
    entity_id,
    digest(
        convert_to(
            E'$NICHOLAS_PASSWORD' || upper(encode(salt_data.salt_bytes, 'hex')),
            'UTF8'
        ),
        'sha256'
    ),
    salt_data.salt_bytes,
    CURRENT_TIMESTAMP
FROM guacamole_entity, salt_data 
WHERE name = 'nicholas' AND type = 'USER';
```

### Key Implementation Details
- **PostgreSQL E String Literals**: Using `E'$VARIABLE'` syntax properly handles special characters after bash variable expansion
- **Salt Format**: 32 random bytes converted to uppercase hexadecimal before concatenation
- **Fixed Salt**: Used deterministic salt for reproducible declarative setup
- **Proper Escaping**: E string literal prefix ensures correct handling of special characters
