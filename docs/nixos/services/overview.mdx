## 🎯 Services Overview

This section covers all web services, monitoring systems, and applications running across the infrastructure, including their configuration, management, and integration.

### 🏗️ Service Architecture

```
Internet
    ↓
[Cloudflare DNS]
    ↓ (HTTPS)
[Traefik Reverse Proxy]
    ├─ Static Routes → Services
    └─ Protected Routes
           ↓
    [OAuth2-Proxy]
           ↓
    [Keycloak OIDC]
           ↓
    [Google IDP + Group Check]
           ↓ (Authorized)
    [Upstream Service]

Hosts:
┌─────────────┬─────────────┬─────────────┐
│    misc     │   jimmich   │    cmos     │
│  (Primary)  │  (Storage)  │ (Desktop)   │
│  - Traefik  │  - Backup   │  - Gaming   │
│  - Keycloak │  - Files    │  - AI/GPU   │
│  - Services │             │             │
└─────────────┴─────────────┴─────────────┘
```

### ⏰ Notification Services

| Service | Host | Purpose | Status | Documentation |
|---------|------|---------|--------|---------------|
| **[Timer Notifications](timer-notifications.md)** | cmos + GCP | Cross-device timer notifications with FCM + RTDB | ✅ Active | [Overview](timer-notifications.md), [Cloud Functions](timer-notifications-cloud-functions.md) |

**Component Stack**:
- **CLI Timer** (`t` command): Cloud-based timer scheduling with Google Cloud Tasks
- **Cloud Functions**: `schedule_timer`, `send_notification`, `dismiss_timer` (Python, GCP)
- **Firebase Realtime Database**: State storage with locked-down security (Admin SDK only)
- **Phone Notifications**: FCM push (timer_complete + timer_dismissed types)
- **Desktop Notifications**: KDE D-Bus notifications with looping alarm sound

**Key Features**:
- **Bidirectional sync**: Dismiss on phone → KDE auto-dismisses (RTDB listener), dismiss on KDE → phone auto-dismisses (FCM push)
- **Looping alarm**: `message-new-instant.oga` plays until dismissed
- **Click-to-dismiss**: Click anywhere on KDE notification (D-Bus default action)
- **TUI Management**: `t list` shows all active timers with live countdown
- **Cloud-based**: Timers survive terminal close and system reboots
- **Secured RTDB**: `.read: false, .write: false` - Admin SDK only access
- **Cost**: &lt; 1% of GCP free tier (virtually free)

**Technical Implementation**:
- **Scheduling**: Google Cloud Tasks with delayed HTTP execution
- **State Management**: Firebase RTDB (`timers/{id}`) with Admin SDK access only
- **Desktop Service**: systemd user service with D-Bus + dbus-monitor for signals
- **Phone Sync**: FCM push with `type: timer_dismissed` when dismissed from desktop
- **Desktop Sync**: RTDB listener detects phone dismissals via status change
- **Authentication**: Firebase Admin SDK (bypasses RTDB security rules)
- **Secrets**: Encrypted via agenix (`firebase-service-account.age`, `fcm-device-phone-token.age`)

### 🏠 Service Dashboards

**Security Dashboard** (security.bedrosn.com):
- Service protection status overview
- Authentication status for all services
- Security posture scores
- Quick access to all services

**[Security Dashboard](security-dashboard.md)** (https://security.bedrosn.com):
- Real-time service inventory and authentication status
- **Security posture scores** - Mozilla Observatory-based header analysis (0-100)
- User access control visibility for Keycloak-protected services
- Background caching for sub-second response times (&lt;0.2s)
- OAuth2 authentication via Keycloak with Google IDP

**Features**:
- **4-Column Layout**: Service, Authentication, Users, Security Posture
- **Security Scoring**: Real-time HTTP header analysis with detailed breakdown
  - HSTS, CSP, X-Frame-Options, X-Content-Type-Options compliance
  - Visual indicators (🟢 Present, 🟡 Default, 🔴 Missing)
  - Score deductions for missing critical headers
- **Live API Queries**: Real-time data from Keycloak API and HTTP header fetching
- **User Access Display**: Lists authorized users for each Keycloak-protected service
- **Background Caching**: 5-minute refresh cycle (4.7s → 0.19s load time)

**Technical Implementation**:
- **Backend**: FastAPI (Python) on port 8091 with async header fetching
- **Authentication**: OAuth2-Proxy (port 4182) with `security-dashboard-access` group
- **Security**: Comprehensive systemd hardening (see [hardening guide](../security/security-dashboard-hardening.md))
- **Performance**: Parallel HTTP requests with asyncio.gather(), global cache dictionary

## 🛡️ Network Security

### 🔍 Intrusion Detection & Prevention

**[Clear NDR Community](../guides/suricata.md)** (suricata.bedrosn.com):
- Suricata-based network detection and response platform
- Real-time network traffic analysis and threat detection
- Advanced packet capture and event correlation
- Comprehensive rule management and threat hunting

**Component Stack**:
- **Suricata**: IDS/IPS engine monitoring network interface enp3s0f0
- **Scirius**: Web-based ruleset management and hunting interface
- **EveBox**: Modern event viewer for real-time alert browsing
- **OpenSearch**: Log storage and analytics (2GB RAM allocation)
- **Arkime**: Full packet capture and PCAP browsing
- **PostgreSQL + RabbitMQ**: Backend services

**Key Features**:
- **Threat Detection**: ET Open ruleset with automatic daily updates
- **Network Monitoring**: Full visibility into network traffic patterns
- **Event Analysis**: Filterable alerts with detailed metadata
- **Rule Management**: Enable/disable rulesets via web UI
- **Packet Capture**: Full PCAP storage and analysis
- **Dashboards**: OpenSearch visualizations for traffic analysis

**Technical Implementation**:
- **Deployment**: Docker Compose stack (13+ containers)
- **Authentication**: OAuth2-proxy (port 8095) + Scirius login (clearndr/clearndr)
- **Access Control**: Keycloak group `suricata-access` required
- **Ports**: 8096 (HTTPS), 8097 (HTTP) - remapped to avoid Traefik conflict
- **Management**: stamusctl binary at `/var/lib/clear-ndr/bin/`
- **Helper Commands**: `clear-ndr-status`, `clear-ndr-logs`, `clear-ndr-restart`

**Hardware Considerations**:
- **Host**: misc (i5-4260U, 15GB RAM, 816GB disk)
- **Resource Usage**: Borderline for Clear NDR (OpenSearch 2GB + 11 other containers)
- **Network**: Monitoring enp3s0f0 interface

## 🔐 Service Security

### 🛡️ Authentication Integration

**Modern Stack (Keycloak + OAuth2-Proxy)**:
- **Keycloak**: Centralized identity provider with Google IDP federation
  - Whitelist-based user pre-creation
  - First Broker Login flow (no auto-registration)
  - Declarative group management via `keycloak-user-config.age`
  - OIDC tokens with groups claim
- **OAuth2-Proxy**: Service-level authorization proxy
  - OIDC client for Keycloak
  - Group-based access control (`--allowed-group`)
  - Per-service authorization (e.g., `docs-access`, `glances-access`)
  - Session management with secure cookies
- **Services Using**: Glances, Memos, NixDocs, Security Dashboard, SABnzbd, Radarr, Sonarr, PrivateBin

**Benefits of Keycloak Stack**:
- ✅ **Declarative**: Groups auto-created from user config
- ✅ **Scalable**: Add services by updating one config file
- ✅ **Fine-grained**: Per-service group membership
- ✅ **Auditable**: All access decisions in Keycloak logs

### 🌐 Network Security

**Traefik Security**:
- TLS termination with Let's Encrypt
- Security headers (HSTS, CSP, etc.)
- Rate limiting and DDoS protection
- Geographic access control

**Service Isolation**:
- **Systemd**: Service sandboxing
- **Network**: Isolated service networks
- **Storage**: Separate data directories
- **Users**: Dedicated service users

## 🚨 Troubleshooting Guide

### 🔧 Common Issues

**Service Won't Start**:
```bash
# Check service status
systemctl status <service-name>

# Check service logs
journalctl -u <service-name> -n 50

# Check dependencies
systemctl list-dependencies <service-name> --failed
```

**Authentication Problems**:
```bash
# Check Keycloak status
systemctl status keycloak

# Check OAuth2 proxy services
systemctl status oauth2-proxy-*

# Test Keycloak health
curl -I https://keycloak.bedrosn.com/realms/web
```

**Performance Issues**:
```bash
# Check resource usage
htop
iostat -x 1

# Check network connectivity
ping target-host
curl -w "@curl-format.txt" https://service.bedrosn.com
```

### 🆘 Emergency Procedures

**Service Outage**:
1. **Immediate**: Check service status and logs
2. **Assessment**: Determine scope and impact
3. **Communication**: Notify users if needed
4. **Resolution**: Restart or reconfigure service
5. **Follow-up**: Document incident and prevention

**Performance Degradation**:
1. **Monitoring**: Identify performance bottlenecks
2. **Resource Check**: Verify CPU, memory, disk usage
3. **Network**: Check network connectivity and latency
4. **Optimization**: Apply performance tuning
5. **Verification**: Confirm performance restoration
