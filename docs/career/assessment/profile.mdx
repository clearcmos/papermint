## The Code Literacy Gap

You've stated that you can't read code syntax directly. This is a real limitation, but it's narrower than it sounds:

- You understand APIs, authentication patterns, rate limiting, pagination, checkpoint/resume, error handling — these are the hard concepts. Syntax is the easy part.
- You've been immersed in Python for 2+ years through AI-assisted development. You likely understand more than you give yourself credit for.
- The gap is closable: spend a few weeks intentionally reading (not writing) the scripts Claude produces. Focus on understanding control flow, not memorizing syntax.

**Why it matters:** When something breaks at 2am and your AI tool is slow or unavailable, you need to be able to triage your own systems. When someone asks "is this safe to run against production Okta?" you need to answer from understanding, not trust. Closing this gap makes your credibility bulletproof.

## Where This Matters in Hiring

### Processes I Will Crush

- **Take-home assignments** - "build us a proof of concept over the weekend." Give me a problem, time, and Claude. This is my strongest format.
- **Architecture discussions** - "walk us through how you'd design X." I can do this cold, at a whiteboard, with confidence.
- **Portfolio reviews** - "show us what you've built." The briefs and repos speak for themselves.
- **System design rounds** - "how would you connect these three systems?" This is literally what I do every day.
- **Behavioral/STAR format** - "tell me about a time you solved a hard problem." I have real stories with real depth.

### Processes That Will Expose Me

- **Live coding rounds** - any format where I type code while someone watches. Pair programming interviews. HackerRank screens.
- **Code review exercises** - "here's a PR, what's wrong with it?" I'd catch architecture issues but miss syntax/language-specific bugs.
- **Whiteboard coding** - "implement this function." Even simple ones - I'll get the logic right but the syntax will be off.
- **CS fundamentals quizzes** - Big O, data structures, algorithm complexity. Not my world.

### Processes That Could Go Either Way

- **Technical deep-dives** - depends on topic. Identity, cloud, automation, Terraform patterns - I'm strong. Language internals, framework-specific gotchas - I'm weak.
- **Pair programming (non-interview)** - on the job, with my own setup and Claude available, I'm productive. In an interview setting without AI, I'm not.

## Target Company Profile

**Best fit:**
- Startups and growth-stage companies that value output over process
- Companies that embrace AI-assisted development
- Roles focused on infrastructure, automation, identity, platform operations
- Take-home or portfolio-based hiring processes
- Teams that need someone who can own IT/security/automation end-to-end

**Avoid:**
- Companies with mandatory live coding rounds (unless you can negotiate an alternative)
- Roles that require daily pair programming with other devs
- Pure software engineering positions (backend dev, full-stack dev)
- Companies that view AI assistance negatively

**Ask during screening:**
- "What does your technical evaluation process look like?"
- "Do you offer take-home assignments as an alternative to live coding?"
- "How does your team feel about AI-assisted development?"

If the answer to any of these is wrong, it's better to know early than to waste time in a process designed to test something you're not.
