:::: section
::: pitch
The vendor provided **no API and no bulk export**. The only option was a web portal designed for one-document-at-a-time manual downloads. I reverse-engineered the entire platform - authentication, navigation, document retrieval - and built a production extraction pipeline that pulled 4,000+ sensitive employee documents on deadline, with checkpoint/resume, corruption detection, and a full audit trail.

## Key Technical Decisions

:::::: card
### Reverse Engineering vs. Waiting for Vendor

::: decision
[Problem]{.decision-label .problem} [No API. Vendor offered no bulk export. Manual download of 4,000+ documents would take weeks. Hard deadline approaching.]{.decision-text}
:::

::: decision
[Choice]{.decision-label .choice} [Reverse-engineered the web portal using HAR file analysis, JavaScript parameter extraction, and ASP.NET GridView form parsing.]{.decision-text}
:::

::: decision
[Result]{.decision-label .result} [Full extraction completed on deadline. Alternative was hiring temps to click through a portal for weeks.]{.decision-text}
:::
::::::

:::::: card
### Checkpoint/Resume System

::: decision
[Problem]{.decision-label .problem} [Multi-hour export across thousands of employees. Network interruptions, session timeouts, and auth expiration are inevitable.]{.decision-text}
:::

::: decision
[Choice]{.decision-label .choice} [JSON checkpoint that auto-saves every 60 seconds. Tracks every downloaded document by URL, every processed employee, current page position, and failed documents with retry counts.]{.decision-text}
:::

::: decision
[Result]{.decision-label .result} [Export survived multiple interruptions and session timeouts without losing a single document. Resume from exact point of failure.]{.decision-text}
:::
::::::

:::::: card
### Post-Export Integrity Pipeline

::: decision
[Problem]{.decision-label .problem} [Sensitive payroll data (tax forms, employment records) entering a new system. Can\'t afford corrupted or duplicate files in the migration.]{.decision-text}
:::

::: decision
[Choice]{.decision-label .choice} [Three-stage post-processing: PDF corruption scanning (PyPDF2, multiprocessing), SHA-256 content deduplication, and CSV audit reporting.]{.decision-text}
:::

::: decision
[Result]{.decision-label .result} [Caught corrupted files before they entered the new system. Identified redundant documents, reducing migration scope. Full audit trail for compliance.]{.decision-text}
:::
::::::

:::::: card
### Dynamic Field Detection

::: decision
[Problem]{.decision-label .problem} [The HRIS system\'s employee fields vary by company configuration. Hardcoding column names would break if the vendor changed their UI.]{.decision-text}
:::

::: decision
[Choice]{.decision-label .choice} [Parse the HTML table header dynamically. Build the CSV export from whatever columns exist at runtime.]{.decision-text}
:::

::: decision
[Result]{.decision-label .result} [Extracted 47+ employee fields without hardcoding a single column name. Resilient to UI changes.]{.decision-text}

## Talking Points

::::::::::: {.card .stories}
### \"Tell me about this project\"

:::: talk-point
::: answer
We were decommissioning an HRIS platform. 4,000+ sensitive employee documents - tax forms, payroll, employment records - needed to come out. The vendor had no API, no bulk export, and no interest in helping. I opened browser dev tools, captured the network traffic, and **reverse-engineered the entire platform**. Built a production pipeline with checkpoint/resume, corruption detection, and deduplication. All documents extracted on deadline. The alternative was hiring temps to click through a web portal for weeks.
:::
::::

### \"What was the hardest part?\"

:::: talk-point
::: answer
The authentication chain. Five layers deep - SSO, SAML, WS-Federation, MFA (push/SMS/TOTP), and session cookies. Each one hands off to the next with different protocols. I also had to deal with JavaScript popup URLs that weren\'t real HTTP links (regex extraction to convert them), and CloudFlare email obfuscation that required XOR decryption to decode employee emails.
:::
::::

### \"How did you handle reliability?\"

:::: talk-point
::: answer
The export runs for hours. Sessions expire, networks drop. I built a **checkpoint system that auto-saves every 60 seconds** - tracks every downloaded document, every processed employee, current page, and failed items with retry counts. If it dies at document 3,500, it resumes from 3,501. After extraction, a corruption scanner validates every PDF and a deduplicator catches identical files with different names.
:::
::::

### \"What did you learn?\"

:::: talk-point
::: answer
That **when a vendor says \"no API,\" it\'s not a dead end - it\'s a reverse-engineering problem.** The web portal is an API, just an undocumented one. Also learned to think about the full pipeline early - extraction is only step one. Integrity validation and audit trails are what make the difference between \"we got the files\" and \"we can prove we got the right files.\"
