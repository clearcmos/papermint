## Key Technical Decisions

:::::: card
### Thread Reconstruction via Topological Sort

::: decision
[Problem]{.decision-label .problem} [Google Vault exports MBOX files, but Gmail\'s threading depends on insertion order. If you restore a reply before its parent, it creates a new thread instead of joining the existing one.]{.decision-text}
:::

::: decision
[Choice]{.decision-label .choice} [Parse RFC2822 headers (In-Reply-To, References) to build a dependency graph, then topologically sort messages so parents always go in first. Rewrite Message-ID mappings on the fly as Gmail assigns new IDs.]{.decision-text}
:::

::: decision
[Result]{.decision-label .result} [Restored emails land in the correct conversation threads. Users see their mailbox as it was, not a mess of orphaned messages.]{.decision-text}
:::
::::::

:::::: card
### Thread-Local Service Instances for Concurrency

::: decision
[Problem]{.decision-label .problem} [Google API client objects are not thread-safe. Sharing them across ThreadPoolExecutor workers causes intermittent request failures and corrupted state.]{.decision-text}
:::

::: decision
[Choice]{.decision-label .choice} [Each worker thread creates its own API service instance using thread-local storage. Combined with a custom token bucket rate limiter to stay under Gmail\'s 15,000 quota units/minute.]{.decision-text}
:::

::: decision
[Result]{.decision-label .result} [Concurrent processing without shared-state bugs. Rate limiter prevents quota exhaustion without sacrificing throughput.]{.decision-text}
:::
::::::

:::::: card
### BigQuery-Backed Data Lifecycle

::: decision
[Problem]{.decision-label .problem} [Deciding which files to archive or delete based on \"nobody uses this\" requires actual usage data, not just modified dates. Drive\'s \"last viewed\" isn\'t exposed in the standard API.]{.decision-text}
:::

::: decision
[Choice]{.decision-label .choice} [Query BigQuery audit logs for last-access timestamps, cross-reference with Drive API metadata, and combine into a single Sheets report with auto-generated outreach to owners.]{.decision-text}
:::

::: decision
[Result]{.decision-label .result} [Cleanup decisions based on real access data, not guesswork. Owners get personalized emails before anything is removed.]{.decision-text}
:::
::::::

:::::: card
### Multi-Credential Impersonation

::: decision
[Problem]{.decision-label .problem} [Transferring Drive ownership requires acting as the current owner (to grant), while restoring permissions requires admin access. One credential can\'t do both.]{.decision-text}
:::

::: decision
[Choice]{.decision-label .choice} [Domain-wide delegation with per-operation impersonation. Same service account, different impersonated users depending on the operation. Credentials retrieved from secret manager at runtime.]{.decision-text}
:::

::: decision
[Result]{.decision-label .result} [Clean separation of privilege. No admin credentials stored anywhere. Audit logs show exactly which user was impersonated for each action.]{.decision-text}
:::

## Talking Points

::::::::::: {.card .stories}
### \"Tell me about this project\"

:::: talk-point
::: answer
I built a suite of 28+ tools that automate the operational side of Google Workspace - the stuff the admin console doesn\'t do. The biggest one is an **email restoration system that preserves conversation threading**. When you export email from Google Vault and re-import it, you normally lose all threading - every message becomes its own conversation. I built a pipeline that parses RFC2822 headers, topologically sorts messages so parents insert before replies, and rewrites Message-IDs on the fly. The result is a restored mailbox that looks exactly like the original.
:::
::::

### \"What was the hardest part?\"

:::: talk-point
::: answer
Concurrency with the Google APIs. The client objects aren\'t thread-safe, so sharing them across threads causes silent failures. I had to use **thread-local storage for API service instances** combined with a custom token bucket rate limiter to stay under quota. The archive cleanup tool was especially tricky - it combines Gmail, Drive, BigQuery, and Sheets APIs in one workflow, each with different rate limits, and needs to handle quota errors, precondition failures, and network timeouts differently.
:::
::::

### \"What problem does this solve that the admin console doesn\'t?\"

:::: talk-point
::: answer
Most of these. The admin console can transfer file ownership one file at a time. My tool walks entire folder trees, snapshots all permissions, transfers ownership, and restores every ACL - user, group, and domain-level. The archive cleanup uses BigQuery for real access data, not just modified dates. The Vault restoration preserves threading. These are the gaps between what Google provides and what IT operations actually needs at scale.
:::
::::

### \"What did you learn?\"

:::: talk-point
::: answer
That **the hard part of API automation isn\'t calling the API - it\'s handling the failure modes**. Rate limits, quota exhaustion, precondition errors, thread safety, session expiration. The happy path is 20% of the code. The other 80% is making it reliable enough that you can kick off a multi-hour operation and walk away. Also learned that BigQuery is underused for IT operations - most teams don\'t realize they have detailed access logs sitting there.
